# セキュリティレビュー

すべてのコードがセキュリティのベストプラクティスに従い、脆弱性を作り込まないことを保証するためのチェックリストとパターン。詳細は `references/` 配下を参照する。

## 起動タイミング

- 認証または認可を実装するとき
- ユーザー入力またはファイルアップロードを扱うとき
- 新しい API エンドポイントを作成するとき
- シークレットや認証情報を扱うとき
- 決済機能・機微データを実装するとき
- サードパーティ API を統合するとき
- 本番リリース前のセキュリティ確認

## 主要原則

- **シークレットはコードに置かない**: 必ず環境変数か Secrets Manager 経由で注入する
- **すべての入力をバリデートする**: スキーマ駆動（Zod / Pydantic 等）。境界で `parse` してから内部に流す
- **クエリは必ずパラメータ化**: 文字列連結による SQL 構築は厳禁
- **トークンは httpOnly クッキー**: `localStorage` は XSS で抜かれる
- **機微操作の前に認可チェック**: 認証が通っても権限がなければ 403 を返す
- **エラーメッセージは汎用化**: スタックトレース・SQL エラーを直接返さない。詳細はサーバーログだけに残す
- **レート制限はデフォルト ON**: 公開エンドポイントは無制限リクエストを許さない
- **依存関係を最新に**: `npm audit` クリーン、ロックファイル commit、Dependabot 有効

## デプロイ前セキュリティチェックリスト

本番デプロイの前に必ず:

- [ ] **シークレット**: ハードコードされたシークレットがなく、すべて環境変数にある
- [ ] **入力バリデーション**: すべてのユーザー入力がバリデートされている
- [ ] **SQL インジェクション**: すべてのクエリがパラメータ化されている
- [ ] **XSS**: ユーザーコンテンツがサニタイズされている、CSP 設定済み
- [ ] **CSRF**: 状態変更操作に保護がある、SameSite=Strict
- [ ] **認証**: 適切なトークン処理（httpOnly Cookie）
- [ ] **認可**: ロールチェックが配置されている、RLS 有効
- [ ] **レート制限**: すべてのエンドポイントで有効
- [ ] **HTTPS**: 本番で強制されている
- [ ] **セキュリティヘッダ**: CSP、X-Frame-Options、HSTS 設定済み
- [ ] **エラーハンドリング**: エラーに機微データがない、スタックトレース非露出
- [ ] **ロギング**: 機微データがログに残っていない
- [ ] **依存関係**: 最新で脆弱性なし、ロックファイル commit 済み
- [ ] **CORS**: 許可されたオリジンのみに構成
- [ ] **ファイルアップロード**: バリデート済み（サイズ、MIME、拡張子）
