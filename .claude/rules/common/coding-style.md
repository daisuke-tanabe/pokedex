# コーディングスタイル

言語・フレームワークに依存しない、コード全般の書き方の原則をまとめる。

## 基本原則

### 可読性最優先

- 明確な変数名と関数名を使う
- コメントよりも自己説明的なコードを優先する
- 一貫したフォーマットを保つ

### KISS（Keep It Simple）

- 実際に動く最もシンプルな解決策を選ぶ
- 早すぎる最適化を避ける
- 賢さや短さよりも明快さを最適化する

### DRY（Don't Repeat Yourself）

- 繰り返されるロジックは共有関数・ユーティリティに抽出する
- コピーペーストによる実装ドリフトを避ける
- 反復が現実のものになったら抽象化する。投機的な抽象化はしない

### YAGNI（You Aren't Gonna Need It）

- 必要になる前に機能や抽象化を作らない
- 投機的な汎用化を避ける
- シンプルに始め、現実に圧力がかかってからリファクタリングする

## 命名規則

- 真偽値: `is` / `has` / `should` / `can` のいずれかをプレフィックスとする
- 真偽値の名前に否定形 (`not`) を含めない (例: `isNotActive` ではなく `isInactive`)

## ファイル構成

多くの小さなファイル > 少ない大きなファイル：
- 高凝集・低結合
- 通常200〜400行、最大800行
- 大きなモジュールからユーティリティを抽出する
- タイプ別ではなく、機能・ドメイン別に整理する

## エラーハンドリング

常に包括的にエラーを処理する：
- 各レベルで明示的にエラーを処理する
- UI向けコードではユーザーフレンドリーなエラーメッセージを提供する
- サーバー側では詳細なエラーコンテキストをログに残す
- エラーを無視して握り潰さない

## 入力バリデーション

常にシステム境界でバリデーションを行う：
- 処理前にすべてのユーザー入力をバリデートする
- 利用可能な場合はスキーマベースのバリデーションを使用する
- 明確なエラーメッセージでフェイルファストする
- 外部データを信頼しない（APIレスポンス、ユーザー入力、ファイルコンテンツ）

## 避けるべきコードスメル

### 深いネスト

ロジックが積み重なり始めたら、ネストした条件分岐より早期リターンを優先する。

### マジックナンバー

意味のある閾値・遅延・上限には名前付き定数を使う。

### 長い関数

大きな関数は責務が明確な小さな単位に分割する。

### ネストした三項演算子

三項演算子のネストは可読性を著しく損なう。
複数条件の分岐は switch 文・if/else・早期リターンを使う。

### 不要な if 文

条件によって boolean 値や同じ呼び出しを切り替えるだけなら、if 文は不要。
値を直接渡すか、三項演算子を式として使う。

## 副作用を避ける

関数は引数と戻り値だけで完結させる。

- 関数のスコープ外の変数を変更しない
- 引数のオブジェクト・配列を変更しない（新しいインスタンスを返す）
- 副作用が必要な場合はその意図を明示する

## リファクタリングの原則

既存コードを書き換える際は、振る舞いを変えずに書き方だけを変える。

- 外部から観測される動作・出力・型インターフェースを維持する
- 過剰な単純化で可読性・保守性を犠牲にしない
- 有用な抽象や責務分離を、行数削減のために潰さない
- 賢すぎる解より、素直で読める解を選ぶ
