# コーディングスタイル

## イミュータビリティ（重要）

常に新しいオブジェクトを生成し、既存のオブジェクトを変更してはならない：

```
// 疑似コード
誤り：  modify(original, field, value) → originalをその場で変更する
正解： update(original, field, value) → 変更を加えた新しいコピーを返す
```

理由：イミュータブルなデータは隠れた副作用を防ぎ、デバッグを容易にし、安全な並行処理を可能にする。

## 基本原則

### 可読性最優先

- コードは書かれるよりも読まれる回数のほうが多い
- 明確な変数名と関数名を使う
- コメントよりも自己説明的なコードを優先する
- 一貫したフォーマットを保つ

### KISS（Keep It Simple）

- 実際に動く最もシンプルな解決策を選ぶ
- 早すぎる最適化を避ける
- 賢さよりも明快さを最適化する

### DRY（Don't Repeat Yourself）

- 繰り返されるロジックは共有関数・ユーティリティに抽出する
- コピーペーストによる実装ドリフトを避ける
- 反復が現実のものになったら抽象化する。投機的な抽象化はしない

### YAGNI（You Aren't Gonna Need It）

- 必要になる前に機能や抽象化を作らない
- 投機的な汎用化を避ける
- シンプルに始め、現実に圧力がかかってからリファクタリングする

## ファイル構成

多くの小さなファイル > 少ない大きなファイル：
- 高凝集・低結合
- 通常200〜400行、最大800行
- 大きなモジュールからユーティリティを抽出する
- タイプ別ではなく、機能・ドメイン別に整理する

## エラーハンドリング

常に包括的にエラーを処理する：
- 各レベルで明示的にエラーを処理する
- UI向けコードではユーザーフレンドリーなエラーメッセージを提供する
- サーバー側では詳細なエラーコンテキストをログに残す
- エラーを無視して握り潰さない

## 入力バリデーション

常にシステム境界でバリデーションを行う：
- 処理前にすべてのユーザー入力をバリデートする
- 利用可能な場合はスキーマベースのバリデーションを使用する
- 明確なエラーメッセージでフェイルファストする
- 外部データを信頼しない（APIレスポンス、ユーザー入力、ファイルコンテンツ）

## 命名規則

- 変数・関数: `camelCase` で説明的な名前
- 真偽値: `is` / `has` / `should` / `can` のいずれかをプレフィックスとする
- インターフェース・型・コンポーネント: `PascalCase`
- 定数: `UPPER_SNAKE_CASE`
- カスタムフック: `use` プレフィックス + `camelCase`

## 避けるべきコードスメル

### 深いネスト

ロジックが積み重なり始めたら、ネストした条件分岐より早期リターンを優先する。

### マジックナンバー

意味のある閾値・遅延・上限には名前付き定数を使う。

### 長い関数

大きな関数は責務が明確な小さな単位に分割する。

## コード品質チェックリスト

作業完了前に確認：
- [ ] コードが読みやすく、適切な命名がされている
- [ ] 関数が小さい（50行未満）
- [ ] ファイルが集中している（800行未満）
- [ ] 深いネストがない（4レベル超なし）
- [ ] 適切なエラーハンドリング
- [ ] ハードコードされた値がない（定数またはconfigを使用）
- [ ] ミューテーションがない（イミュータブルなパターンを使用）
