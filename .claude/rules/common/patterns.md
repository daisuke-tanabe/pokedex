# 共通パターン

## デザインパターン

### スタートポロジー

同じレイヤーのモジュールは互いに直接依存してはならない。各モジュールはコア（コンポジションルート・ホスト・共有インターフェース）を通じてのみ通信する:
- 同一カテゴリのモジュールは互いをインポートしない（例: ルート同士、サービス同士、リポジトリ同士、ハンドラー同士、ストア同士）
- すべての結合はコンポジションルートで依存性注入によって行う

これにより依存グラフがハブ・アンド・スポーク構造を維持し、N² の結合や連鎖的な破壊を防ぐ。

### リポジトリパターン

一貫したインターフェースの背後にデータアクセスをカプセル化する：
- 標準操作を定義する：findAll, findById, create, update, delete
- 具体的な実装がストレージの詳細を処理する（データベース、API、ファイルなど）
- ビジネスロジックはストレージメカニズムではなく、抽象インターフェースに依存する
- データソースの簡単な切り替えを可能にし、モックによるテストを簡素化する

### APIレスポンスフォーマット

すべてのAPIレスポンスに一貫したエンベロープを使用する：
- 成功/ステータスインジケーターを含める
- データペイロードを含める（エラー時はnullable）
- エラーメッセージフィールドを含める（成功時はnullable）
- ページネーションレスポンスにメタデータを含める（total, page, limit）

### REST API エンドポイント規約

HTTP メソッドとパス命名は標準的な REST 規約に従う：

```
GET    /api/<resources>          # コレクションを一覧
GET    /api/<resources>/:id      # 特定リソースを取得
POST   /api/<resources>          # 新規リソースを作成
PUT    /api/<resources>/:id      # リソースを更新（フル置換）
PATCH  /api/<resources>/:id      # リソースを更新（部分）
DELETE /api/<resources>/:id      # リソースを削除

# フィルタリング・ページネーション用のクエリパラメータ
GET /api/<resources>?status=active&limit=10&offset=0
```

- リソース名は複数形・kebab-case（複合語）
- 動詞をパスに含めない（HTTP メソッドが動詞の役割）
- ネストはオーナーシップ表現にとどめる（`/api/users/:id/orders` 等）
