# Refactor Clean

ステップごとにテスト検証を挟みながら、デッドコードを安全に特定・削除する。

## Step 1: デッドコード検出

プロジェクトタイプに応じた解析ツールを実行する：

| ツール | 検出対象 | コマンド |
|------|--------------|---------|
| knip | 未使用 export、ファイル、依存 | `npx knip` |
| depcheck | 未使用 npm 依存 | `npx depcheck` |
| ts-prune | 未使用 TypeScript export | `npx ts-prune` |
| vulture | 未使用 Python コード | `vulture src/` |
| deadcode | 未使用 Go コード | `deadcode ./...` |
| cargo-udeps | 未使用 Rust 依存 | `cargo +nightly udeps` |

ツールが利用できない場合は Grep で import が0件の export を探す：
```
# export を探し、それがどこにも import されていないかを確認する
```

## Step 2: 結果のカテゴリ分け

安全度ティアで分類する：

| ティア | 例 | アクション |
|------|----------|--------|
| **SAFE** | 未使用ユーティリティ、テストヘルパー、内部関数 | 自信を持って削除 |
| **CAUTION** | コンポーネント、APIルート、middleware | 動的import や外部利用者がないことを検証 |
| **DANGER** | 設定ファイル、エントリポイント、型定義 | 触る前に詳細調査 |

## Step 3: 安全な削除ループ

各 SAFE 項目について：

1. **全テスト実行** — ベースラインを確立する（全グリーン）
2. **デッドコード削除** — Edit ツールで的確に削除する
3. **テスト再実行** — 何も壊れていないことを検証
4. **テスト失敗時** — `git checkout -- <file>` で即時リバートし、その項目はスキップ
5. **テスト成功時** — 次の項目へ進む

## Step 4: CAUTION 項目の取り扱い

CAUTION 項目を削除する前に：
- 動的 import を検索する：`import()`, `require()`, `__import__`
- 文字列参照を検索する：ルート名、設定内のコンポーネント名
- 公開パッケージAPIから export されていないか確認する
- 外部利用者がいないか確認する（公開済みなら依存先をチェック）

## Step 5: 重複の整理

デッドコード削除後、次を探す：
- 類似度80%超のほぼ重複関数 — 1つにマージする
- 冗長な型定義 — 統合する
- 価値のないラッパー関数 — インライン化する
- 不要な再 export — 間接層を取り除く

## Step 6: サマリ

結果を報告する：

```
Dead Code Cleanup
──────────────────────────────
Deleted:   12 unused functions
           3 unused files
           5 unused dependencies
Skipped:   2 items (tests failed)
Saved:     ~450 lines removed
──────────────────────────────
All tests passing PASS:
```

## ルール

- **テスト未実行で削除しない**
- **削除は1件ずつ** — アトミックな変更でロールバックを容易にする
- **不安なら飛ばす** — プロダクション破壊よりデッドコード残存を優先する
- **クリーンと同時にリファクタしない** — 関心を分ける（先にクリーン、次にリファクタ）
