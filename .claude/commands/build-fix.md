# Build and Fix

最小かつ安全な変更で、ビルドエラーと型エラーを段階的に修正する。

## Step 1: ビルドシステムを検出する

プロジェクトのビルドツールを特定し、ビルドを実行する：

| 指標 | ビルドコマンド |
|-----------|---------------|
| `package.json` に `build` script | `npm run build` または `pnpm build` |
| `tsconfig.json` (TypeScriptのみ) | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall -q .` または `mypy .` |

## Step 2: エラーをパースしてグルーピングする

1. ビルドコマンドを実行し、stderrをキャプチャする
2. エラーをファイルパス別にグルーピングする
3. 依存順にソートする（ロジックエラーより先にimport/typeを修正する）
4. 進捗追跡のためエラー総数をカウントする

## Step 3: Fix Loop（一度に1エラー）

各エラーごとに：

1. **ファイルを読む** — Readツールでエラーコンテキスト（エラー周辺10行）を確認する
2. **診断する** — 根本原因を特定する（importの不足、型の誤り、syntax errorなど）
3. **最小限に修正する** — エラーを解決する最小の変更でEditツールを使う
4. **ビルドを再実行する** — エラーが解消され、新しいエラーが導入されていないことを検証する
5. **次へ進む** — 残りのエラーを処理する

## Step 4: ガードレール

次の場合は停止してユーザーに確認する：
- 修正が**解消するエラーより多くのエラー**を導入した
- **同じエラーが3回試行しても残る**（深い問題の可能性）
- 修正が**アーキテクチャ変更**を要する（単なるビルド修正ではない）
- ビルドエラーが**依存関係の不足**に起因する（`npm install`、`cargo add` 等が必要）

## Step 5: サマリ

結果を表示する：
- 修正したエラー（ファイルパス付き）
- 残っているエラー（あれば）
- 新たに導入されたエラー（ゼロであるべき）
- 未解決問題への次のステップの提案

## リカバリ戦略

| 状況 | アクション |
|-----------|--------|
| モジュール/import不足 | パッケージがインストールされているか確認；installコマンドを提案 |
| 型不一致 | 両方の型定義を読み、より狭い型を修正する |
| 循環依存 | importグラフでサイクルを特定し、抽出を提案 |
| バージョン衝突 | `package.json` / `Cargo.toml` のバージョン制約を確認 |
| ビルドツール設定ミス | configファイルを読み、動作するデフォルトと比較 |

安全のため一度に1エラーを修正する。リファクタリングより最小diffを優先する。
