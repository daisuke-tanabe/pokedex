# API 設計パターン

一貫性があり、開発者にやさしい REST API を設計するための規約とベストプラクティス。詳細仕様・実装例は `references/` 配下を参照する。

## 起動タイミング

- 新しい API エンドポイントを設計するとき
- 既存の API 契約をレビューするとき
- ページネーション、フィルタリング、ソートを追加するとき
- API のエラーハンドリングを実装するとき
- API バージョニング戦略を計画するとき
- パブリック／パートナー向け API を構築するとき

## 主要原則

- **リソースは名詞・複数形・kebab-case**: URL に動詞を入れない（HTTP メソッドが動詞の役割）
- **HTTP ステータスコードを意味的に使う**: 成功でも失敗でも 200 を返さない。404 / 422 / 429 などを正しく返す
- **エラーは構造化**: `code` / `message` / `details` を持つ統一フォーマット。内部詳細（スタックトレース、SQL）は漏らさない
- **境界でバリデーション**: 受信データは必ずスキーマ検証してから処理する
- **ページネーションを必須化**: list 系エンドポイントでは無制限取得を許さない（カーソル推奨、オフセットも可）
- **バージョニングは必要になるまでしない**: `/api/v1/` で始め、breaking change を導入する直前に v2 を切る

## API 設計チェックリスト

新エンドポイント出荷前に確認:

- [ ] リソース URL が命名規約に従う（複数形、kebab-case、動詞なし）
- [ ] 正しい HTTP メソッドを使用している（読み取りに GET、作成に POST 等）
- [ ] 適切なステータスコードを返す（すべて 200 ではない）
- [ ] 入力がスキーマで検証されている（Zod、Pydantic、Bean Validation 等）
- [ ] エラーレスポンスがコードとメッセージを持つ統一フォーマットに従う
- [ ] リストエンドポイントにページネーションが実装されている（カーソルまたはオフセット）
- [ ] 認証が必須（または明示的にパブリックとマーク）
- [ ] 認可がチェックされている（ユーザーは自分のリソースのみアクセス可能）
- [ ] レート制限が設定されている
- [ ] レスポンスが内部詳細（スタックトレース、SQL エラー）を漏らさない
- [ ] 既存エンドポイントとの命名一貫性が保たれている（camelCase vs snake_case）
- [ ] ドキュメント化されている（OpenAPI/Swagger 仕様が更新されている）
