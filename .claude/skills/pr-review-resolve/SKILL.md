# pr-review-resolve

GitHub PR に付いたフィードバックを反映する skill。レビュアが付ける重要度ラベルはレビュア間で千差万別 (無ラベル / 任意の英語名 / 数値優先度 / 質問 vs 提案 vs 問題、など)。本 skill は **ラベル名に依存せず、実害ベースで「対応 / 反論 / スルー」を判定**する。

判断が必要な箇所はユーザに確認を取り、対話で進める。

## 入力

- PR 番号 (省略時はカレントブランチに紐づく PR を `gh pr list --head $(git branch --show-current) --json number,title,url --limit 1` で推論)
- 推論失敗時 (空配列 `[]` を返した場合を含む) はユーザに対象 PR を確認する

## ステップ

### 1. フィードバックの収集

GitHub の PR は **PR 本文 / Issue Comment / Review / Review Comment (inline)** が別の API で取れる。それぞれを取得して **指摘の出所と文脈** を明確にする (どこに書かれたか = 解像度を上げる)。

| 出所 | 内容 | 取得方法 |
|---|---|---|
| PR 本文 | PR 作成時の説明・チェックリスト | `gh pr view <N> --json title,body,headRefName,baseRefName` |
| Issue Comment | PR 全体への一般コメント (path/line なし) | `gh pr view <N> --json comments` |
| Review | レビュアの判定 + 本文 (APPROVED / CHANGES_REQUESTED / COMMENTED / DISMISSED ※DISMISSED は管理者により dismiss された change request で実質的に対象外) | `gh pr view <N> --json reviews` |
| Review Comment (inline) | path / line / diff hunk に紐づく。スレッド (リプライ) を含む | `gh api repos/{owner}/{repo}/pulls/<N>/comments` |
| CI ステータス | 自動チェックの結果 | `gh pr view <N> --json statusCheckRollup` (または `gh pr checks <N>`) |
| Review Thread の解決状態 | inline スレッドが resolved か unresolved か (任意) + 各 thread の `id` (resolve mutation の入力) | (下記コード参照) |

`{owner}/{repo}` は `gh repo view --json owner,name --jq '.owner.login + "/" + .name'` で取得できる。

Review Thread の取得 (GraphQL):

```bash
gh api graphql \
  -F owner=<owner> -F repo=<repo> -F number=<N> \
  -f query='query($owner:String!,$repo:String!,$number:Int!){
    repository(owner:$owner,name:$repo){
      pullRequest(number:$number){
        reviewThreads(first:100){
          nodes{
            id
            isResolved
            comments(first:50){ nodes{ id path body author{login} } }
          }
        }
      }
    }
  }'
```

戻り値の `reviewThreads.nodes[].id` がステップ 7 の resolve mutation の `threadId` に対応する。

100 スレッド / 50 コメントを超える大規模 PR では `pageInfo { endCursor hasNextPage }` を取得し `after:$cursor` でページネーションする。

bot / 人手 / 複数レビュアが混在する場合もすべて対象とする。Review Comment の path/line はステップ 2 の「対象箇所」根拠としてそのまま使う。

### 2. 指摘の抽出 + 妥当性検証

各指摘について以下を整理する:

- **出所**: PR 本文 / Issue Comment / Review / Review Comment のどれか
- **重要度**: レビュアが付けたラベル文字列があれば原文ママ、無ければ空欄
- **要旨**: 何を問題視しているか
- **対象箇所**: ファイル / 行番号 / シンボル名 (Review Comment は path/line が API から取れる)
- **妥当性**: 事実か、影響範囲はどこか

妥当性検証はコードを直接読んで行い、以下と照合する:

- 設計文書 / 規約文書 (コントリビューションルール、設計ディレクトリ、ADR 等)
- データ / fixture / テストデータの実態
- 過去の変更履歴 / 議論経緯 (`git log`、過去 PR、Issue)

### 3. 影響範囲の事前検討

修正前に、対象 + 下流への副作用を洗い出して整理する (「修正で何が変わるか」を先に把握する)。省略するとツールチェーン仕様違反や本筋から外れた改善混入のリスクがある。

```markdown
### [影響なし] / 想定内 (確認済)
1. <下流処理> — <理由>

### [要注意] / 確認しながら進める
1. <要注意ポイント> — <対処方針>

### [スコープ外]
- <別の変更単位で扱う事項>
```

### 4. 判定 → ユーザ確認 (停止ポイント)

各指摘を以下の 3 段階で判定する。プロジェクト固有のラベル → 対応原則のマッピングがあればそれを優先する (「判断軸の補足」参照)。

#### [対応] (Resolve)

以下のいずれかなら対応する:

- セキュリティ脆弱性 / 認可漏れ / シークレット漏洩
- データ破損 / データ不整合
- 本番障害 / 性能劣化
- 仕様文書 / 設計文書で明示された振る舞いと実装が乖離 (顕在化していなくても潜在バグ)
- 設計原則違反で保守性に大きく影響
- 修正コストが極小 (1〜5 行) かつ対応箇所が同じコード領域

#### [反論] (Refute)

以下のいずれかなら根拠を示して反論する:

- ツールチェーン / ワークフロー仕様の誤解 (自動反映機構、フレームワーク標準パターン、ビルドツール仕様 など)
- プロジェクトのフロー / ルールの誤解
- 過去議論で別単位として切り出すと合意済みの提案の再提案
- 設計文書 / コードベースの実態と異なる前提

#### [スルー] (Defer)

以下のいずれかなら情報共有のみとし、必要なら別途記録する:

- 軽微なスタイル / 命名 / コメント / 主観的好み
- 将来検討の提案 (現時点で実害がなく別の変更単位で扱うほうが適切)
- 重複指摘 (既に対応済み)
- リファクタ提案で「現状コードに問題はないが、より綺麗にできる」程度

#### 判定提示テンプレート

ユーザに確認する:

```markdown
## レビュー判定

| # | 出所 | 重要度 (レビュア記載) | 判定 | 要旨 | 理由 |
|---|---|---|---|---|---|
| 1 | Review Comment (path:L) | <原文ママ or 空欄> | [対応] | <要旨> | <妥当性検証の根拠> |
| 2 | Review | <原文ママ or 空欄> | [反論] | <要旨> | <反論根拠: 文書 / フロー / 過去経緯> |
| 3 | Issue Comment | <原文ママ or 空欄> | [スルー] | <要旨> | <主観 or 別の変更単位> |

## 影響範囲

(ステップ 3 で整理した内容)

## 修正方針
1. ...

この方針で進めますか?
```

### 5. 修正実装

合意後に実装する。テストが必要な変更 (バグ修正 / 新規ロジック / 振る舞いの変更) は先にテストを追加してから実装する。1〜数行の typo / コメント修正 / 文書修正のような変更はテスト先行を強制しない。

1. テストが必要な変更なら関連既存テストを確認 → 新規 Scenario をテスト追加
2. 実装変更
3. プロジェクトのテスト / 型チェック / lint / フォーマットチェック が全て緑になることを確認

最小差分を心がける。レビューに無い修正は混ぜない (「判断軸の補足: コミットスコープ規律」参照)。

### 6. セルフレビュー再実行

プロジェクトのセルフレビュールールがあればそれに従う。無ければ変更内容に応じて review agent / レビュアを選ぶ。

再レビューで新規指摘があれば **ステップ 2 から再帰**。

### 7. コミット + push + レビューへの応答

#### コミットメッセージ

プロジェクトのコミット規約 (Conventional Commits 等) に合わせる。

```
<type>(<scope>): PR #<N> レビュー反映 (<主要修正の要約>)

## レビュー判定と対応

### <対応 1 タイトル>
<対応内容>

## 追加テスト (なければ省略)
- <test 概要>

## スルー判定 (なければ省略)
- <スルーした指摘と理由>

## 検証
- <型チェック / test / lint / format 結果>
- <セルフレビュー結果>

<プロジェクト規約の attribution があれば末尾に追記>
```

#### Push

```bash
git push origin <head-branch>
```

pre-push hook (format / lint / 型チェック等) がある場合は緑通過を確認する。

#### レビューへの応答

応答方法は指摘の出所で使い分ける:

- **PR 全体への対応サマリ** (通常のレビュー応答): `gh pr comment <N> --body "..."`
- **特定の Review Comment スレッドへのリプライ**: `gh api repos/{owner}/{repo}/pulls/<N>/comments/<comment_id>/replies -f body="..."`
- **Review Thread の解決 (resolve)**: `threadId` はステップ 1 の GraphQL 結果 `reviewThreads.nodes[].id` から取り、変数で渡す:

  ```bash
  gh api graphql \
    -F threadId=<thread-id> \
    -f query='mutation($threadId:ID!){
      resolveReviewThread(input:{threadId:$threadId}){
        thread{ isResolved }
      }
    }'
  ```

通常は PR 全体への対応サマリで足りる。inline で議論が続いている指摘は当該スレッドにリプライする。

PR 全体応答テンプレート:

```markdown
commit `<sha>` での対応内容を整理する。

## 対応一覧

### [対応]
| 指摘 | 対応 |
|---|---|
| <指摘 1> | <対応概要> |

### [反論]
- **<反論対象>**: <根拠を明示>

### [スルー]
- **<スルー対象>**: <理由>

## 追加テスト (なければ省略)
- <概要>

## 検証
- <CI / 型チェック / test / lint / format 結果>
```

#### 完了報告 (ユーザへの最終出力)

```markdown
## レビュー対応完了

- **commit**: `<sha>` (PR #<N> レビュー反映)
- **push**: 完了
- **レビューへの応答**: 投稿済み (<url>)
  - [対応]: <件数>
  - [反論]: <件数>
  - [スルー]: <件数>
```

## 判断軸の補足

ステップ 4 を補強する観点。

### プロジェクト固有ルールとの整合

コントリビューションルール / 開発フロー文書 / 設計判断記録 (ADR) が以下を規定していれば、最初に読み判定の補助にする (= 強い指針として尊重):

- レビュー対応の方針 (どの重要度を対応必須とみなすか、など)
- セルフレビューの方法
- コミット規約 / PR テンプレ
- ブランチ運用 / マージ戦略
- 仕様文書のワークフロー (差分管理 / 自動反映機構の有無)

「重要度ラベル X は対応必須」のようなマッピングがあれば、ステップ 4 の判定 3 段階に **そのままマッピング**する。マッピングが無い / 言及が無いラベルは純粋に実害ベースで判定する。

### 反論の根拠例

ステップ 4 [反論] の典型例:

- **ツールチェーン / ワークフロー仕様の誤解**: 自動反映機構やフロー特性の見落とし
- **フレームワーク標準パターンの誤解**: ライブラリ / フレームワークの慣用への無理解
- **コードベース / データ実態の誤認**: 仕様で明文化されていない振る舞いを「バグ」と指摘、fixture の実態と異なる前提
- **過去議論の見落とし**: 別単位として切り出すと合意済みの提案を再提案

論拠が残る範囲は撤回せず議論を続け、ユーザに最終判断を委ねる。

### 複数レビュアが相反する場合

レビュア A が「X に変えるべき」、レビュア B が「現状維持」のように **相反する** 指摘が来た場合は以下の順で優先する:

1. **CODEOWNERS / コードオーナーの判断**: リポジトリに `CODEOWNERS` ファイルがあり、当該ファイルのオーナーが指摘していればそれを優先
2. **プロジェクトの規約文書**: コントリビューションルール / ADR で方向性が定まっていればそれに従う
3. **PR 作成者の判断**: 1, 2 で決まらない場合は PR 作成者 (= 本 skill 利用者) の判断に委ねる

判定提示時にユーザに「どちらを優先するか」を明示して確認する。

### スルー指摘を例外的に同時対応する条件

スルー判定の指摘でも、以下のいずれかなら同じコミットで対応してよい:

- 対応判定と **同じコード領域** を触る (= 別途修正すると対応後にスルー指摘だけ残る不整合状態になる)
- 修正コストが極小 (1〜5 行)
- 対称性を保つために必要 (例: 一方の入り口は厳密、他方は緩い)

それ以外は **別の変更単位** に積む。

### コミットスコープ規律

レビュー反映 commit に含めてよいのは以下のみ:

- 指摘箇所の修正
- 修正に必要なテスト
- 修正と同領域のスルー指摘 (上記許容ラインを満たすもの)
- 修正に伴うコメント / 文書の整合性回復

レビューに無い refactor / 別バグ修正 / 関連する別作業は **別コミット or 別 PR** に切る。

## ガードレール

- ステップ 4 で必ず止まる: 判定提示後、ユーザの合意なしに修正を始めない
- 反論候補は必ずユーザに伝える: 勝手に「全部対応」して反論機会を奪わない
- 影響範囲の事前検討 (ステップ 3) を省略しない
- レビュアのラベル名に判定を引っ張られない
- プロジェクトルールがあれば最初に読む
- ツールチェーン / ワークフロー仕様を尊重する
- コミットスコープを保つ: レビュー反映 commit に指摘範囲外を混ぜない
- セルフレビュー (ステップ 6) を省略しない
- 反論時は根拠を明示する (文書 / コミットハッシュ / フロー仕様 / 過去議論)
- テストの「既存の skip / failed」を新規発生と混同しない (実行前後で差分を取る)
- 指摘の **出所** (PR 本文 / Issue Comment / Review / Review Comment) を区別する: inline スレッドは path/line と紐づくので「対象箇所」根拠として活用する
