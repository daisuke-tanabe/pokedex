# テスト駆動開発ワークフロー

すべてのコード開発が包括的なテストカバレッジを伴う TDD 原則に従うことを保証する。詳細は `references/` 配下を参照する。

## 起動タイミング

- 新機能や機能の作成
- バグや問題の修正
- 既存コードのリファクタリング
- API エンドポイントの追加
- 新コンポーネントの作成

## 中核原則

### 1. コードより先にテスト

必ずテストを先に書き、テストを通すように実装する（Red → Green → Refactor）。

**1 テストずつ進める**（垂直スライシング）。すべてのテストを先に書いてから全実装に移る「水平スライシング」はアンチパターン。各テストは前サイクルで得た学びに応答する形で書く。詳細は `references/anti-patterns.md`。

### 2. カバレッジ要件

- 最低 **80% のカバレッジ**（ユニット + 統合 + E2E）
- すべてのエッジケースを網羅
- エラーシナリオをテスト
- 境界条件を検証

### 3. テストの種類

| 種類 | 対象 |
|---|---|
| **ユニット** | 個別関数・コンポーネントロジック・純粋関数・ヘルパー |
| **統合** | API エンドポイント・DB 操作・サービス間連携・外部 API |
| **E2E（Playwright）** | 重要なユーザーフロー・完全なワークフロー・UI 操作 |

### 4. Git チェックポイント

各 TDD ステージ後にチェックポイントコミットを作成し、ワークフロー完了まで squash・rewrite しない。**RED 検証コミット → GREEN 検証コミット → リファクタ完了任意コミット** が最小構成。詳細は `references/workflow-steps.md` を参照。

### 5. RED 中はリファクタリングしない

リファクタリングは **すべてのテストが GREEN** の状態でしか行わない（"Never refactor while RED" 規律）。RED 中の構造変更は「失敗の原因」と「設計の改善」が混ざり、原因切り分けが不能になる。

## ワークフロー概要（Red → Green → Refactor）

1. **ユーザージャーニーを書く** — `As a [role], I want to [action], so that [benefit]`
2. **テストケース候補を列挙し、最初の 1 つを書く** — 候補は列挙のみ、コードは 1 テストずつ書く
3. **テストを実行（RED 検証）** — 失敗を確認、`test:` コミット
4. **コードを実装** — 最小限の修正
5. **テスト再実行（GREEN 検証）** — `fix:` コミット
6. **リファクタリング** — 緑を保ったまま品質改善、`refactor:` コミット
7. **カバレッジ検証** — 80%+ 達成

各ステップの詳細・チェックポイントの正確な検証手順は `references/workflow-steps.md`。

## ベストプラクティス（短縮版）

1. **テストを先に書く** — 常に TDD
2. **1 テスト 1 振る舞い** — 単一の振る舞いに集中
3. **説明的なテスト名** — 何をテストしているかが伝わる
4. **Arrange-Act-Assert** — 明確なテスト構造
5. **境界でのみモック** — 自分のコードはモックしない
6. **エッジケースをテスト** — null / undefined / 空 / 巨大
7. **エラー経路をテスト** — ハッピーパスだけでない
8. **テストを高速に保つ** — ユニットテストは 1 件 50ms 未満
9. **クリーンアップ** — 副作用を残さない
10. **カバレッジレポートをレビュー** — ギャップを特定する

## 成功指標

- 80% 以上のコードカバレッジ
- すべてのテストが pass（緑）
- スキップ・無効化されたテストがない
- ユニットテスト実行が 30 秒未満
- E2E テストが重要なユーザーフローを網羅
- テストが本番前にバグを捕捉している
